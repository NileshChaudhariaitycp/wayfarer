# Glossary

Plain-English definitions of terms as they're introduced, in the order we hit them.

| Term | Plain-English definition |
|---|---|
| Maven multi-module reactor | One parent `pom.xml` that lists several sub-projects ("modules") and builds them together in dependency order, sharing version management. |
| BOM (Bill of Materials) | A `pom.xml` with `packaging=pom` that pins compatible versions of a whole family of libraries, so you declare a dependency without specifying its version ‚Äî the BOM decides it. |
| Spring Cloud "release train" | Spring Cloud isn't versioned 1:1 with Spring Boot. Instead, a named release (e.g. `2025.0.x` "Northfields") bundles compatible versions of Eureka, Gateway, Config, OpenFeign, etc. for a specific Spring Boot minor version. |
| Service registry (Eureka) | A directory that services register themselves into on startup ("I'm `flight-service`, I'm at `10.0.0.5:8083`") and query at runtime to find each other, instead of hardcoding IPs/ports. |
| JWT (JSON Web Token) | A signed, self-contained token: a client presents it instead of re-sending credentials on every request. Anyone holding the signing key/secret can verify it wasn't tampered with, without a database lookup. |
| BCrypt | A deliberately slow password-hashing algorithm ‚Äî the slowness is the point, it makes brute-forcing a stolen hash expensive. Never store plaintext or fast-hashed (e.g. plain SHA-256) passwords. |
| "Trust the gateway" | A pattern where only the API gateway validates JWTs; it forwards trusted identity headers (`X-User-Id`, `X-User-Roles`) to backend services, which trust those headers instead of re-parsing tokens. Only safe if backend services are unreachable except through the gateway ‚Äî see ADR 0004. |
| OpenFeign | A declarative REST client: you write a Java interface with annotations describing an HTTP call, and Spring generates the actual HTTP call at runtime ‚Äî including resolving the target service's address via Eureka. |
| `@PreAuthorize` | A Spring Security annotation that guards a method with a permission check (e.g. `hasRole('ADMIN')`), evaluated before the method body runs. Requires `@EnableMethodSecurity`. |
| `@OneToMany` / `@ManyToOne` | JPA relationship mapping ‚Äî one `Hotel` has many `RoomType`s, each `RoomType` belongs to one `Hotel`. `cascade = ALL` + `orphanRemoval = true` means saving/deleting the Hotel saves/deletes its RoomTypes too, since they have no independent lifecycle. |
| Saga pattern | A way to handle a "transaction" that spans multiple services/databases, where no single database transaction can cover all of it. Instead of rollback, each step that already succeeded gets explicitly undone by a compensating action if a later step fails. |
| Orchestration vs. choreography (Saga) | Orchestration: one service (booking-service) directly calls every participant and issues compensating calls on failure ‚Äî a central coordinator. Choreography: services react to each other's published events with no central coordinator ‚Äî needs an event bus (Kafka, Phase 5). |
| Pessimistic write lock (`SELECT ... FOR UPDATE`) | Locks a row so no other transaction can read-and-modify it until the lock-holding transaction commits or rolls back. Only works within the DB connection/transaction that holds the lock ‚Äî can't be held across an HTTP call to another service. |
| Idempotent (in this context) | Calling an operation twice with the same input has the same effect as calling it once. loyalty-service's `earn`/`reverse` check for an existing transaction record before acting, so a retried compensating call can't double-credit or double-refund. |
| Compensating action | The explicit "undo" for a step that already succeeded in a Saga ‚Äî e.g. `releaseSeats` undoes `reserveSeats`, `refund` undoes `authorize`. Not a database rollback; a deliberate, separate operation. |
| Kafka topic / consumer group | A topic is a named event stream (e.g. `booking-events`) that producers publish to and consumers read from. A consumer group is a named set of consumers sharing the work of reading a topic ‚Äî each message goes to one consumer *within* a group, but every group gets its own full copy of the stream. |
| `max.block.ms` | A Kafka producer setting most people never touch: how long `KafkaTemplate.send()` is allowed to block the *calling thread* to resolve topic metadata (partition info) before giving up. Defaults to 60000ms ‚Äî surprising, since `send()` looks like a fire-and-forget async call that returns a `CompletableFuture`. If the broker is unreachable, that initial call can itself hang for a full minute before you even get the future back. |
| Circuit breaker | Wraps a remote call; after enough recent failures (a configurable failure-rate threshold over a sliding window), it "opens" and rejects further calls immediately without even attempting them, for a cooldown period ‚Äî protects a struggling downstream service from being hammered further, and protects the caller from piling up slow/timed-out requests. |
| Idempotency-aware retry | Automatically retrying a failed call is only safe if calling it twice has the same effect as calling it once. In this project, retry is enabled only on `getFlight`/`getHotel` (read-only) and the guarded-idempotent `refund`/loyalty `earn`/`reverse` ‚Äî never on `reserveSeats`/`authorize`, where a retried timeout could double-act on a request that actually succeeded server-side. |
| Trace ID / span ID | A trace ID identifies one logical request as it flows through every service it touches; a span ID identifies one hop within that trace (e.g. the call from booking-service to flight-service). Correct propagation means the *same* trace ID appears in every service's logs for one request, each with a *different* span ID. |
| Trace context propagation | How a trace ID gets from one service to the next ‚Äî typically an HTTP header (`traceparent`) added to the outgoing request and read back out on the receiving end. This doesn't happen automatically for every HTTP client; Feign specifically needs the `feign-micrometer` dependency to hook into it ‚Äî see ADR 0008. |
| Layered jar | Spring Boot's default jar layout split into layers (dependencies, snapshot-dependencies, spring-boot-loader, application) by how often each changes. `java -Djarmode=tools -jar app.jar extract --layers` unpacks them separately, so a Docker image can `COPY` the rarely-changing dependency layers first and the frequently-changing application layer last ‚Äî better build-cache reuse than one monolithic `COPY app.jar`. |
| 12-factor config | Configuring an app entirely through environment variables (not baked-in profile files), so the exact same build artifact runs unmodified in every environment ‚Äî only what's injected around it differs. Spring Boot's relaxed binding maps env vars like `SPRING_DATASOURCE_URL` straight onto `spring.datasource.url`. |
| Jackson default typing (polymorphic typing) | Jackson can embed a type hint (`@class` property, or an array-wrapper form) in serialized JSON so a later deserialization call ‚Äî where the target type is `Object`/generic and otherwise unknown ‚Äî can reconstruct the right concrete class. Powerful but finicky (inconsistent wrapper formats for "non-natural" scalar types like `BigDecimal`) and a known deserialization-gadget security risk if the JSON source isn't fully trusted ‚Äî best avoided when a type-specific serializer is possible instead. |
| Redis cache serializer binding | Configuring a *specific* Jackson `JavaType` (e.g. `List<FlightResponse>`) for one named Spring cache, instead of relying on a generic serializer with embedded type metadata. Since a given `@Cacheable` cache name in this project only ever holds one well-known return type, this sidesteps default typing entirely ‚Äî Jackson is simply told up front what to deserialize into. |
| Zipkin reporter | The component that actually ships completed spans to a Zipkin server over HTTP. `micrometer-tracing-bridge-brave` creates and correlates trace/span IDs (visible in log output) but does nothing to *export* them ‚Äî a separate `zipkin-reporter-brave` dependency is required for spans to ever leave the process. Missing it is a silent failure: sampling and log correlation work fine, so nothing looks obviously broken until you check the Zipkin UI and find it empty. |
