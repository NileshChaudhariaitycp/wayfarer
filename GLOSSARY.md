# Glossary

Plain-English definitions of terms as they're introduced, in the order we hit them.

| Term | Plain-English definition |
|---|---|
| Maven multi-module reactor | One parent `pom.xml` that lists several sub-projects ("modules") and builds them together in dependency order, sharing version management. |
| BOM (Bill of Materials) | A `pom.xml` with `packaging=pom` that pins compatible versions of a whole family of libraries, so you declare a dependency without specifying its version — the BOM decides it. |
| Spring Cloud "release train" | Spring Cloud isn't versioned 1:1 with Spring Boot. Instead, a named release (e.g. `2025.0.x` "Northfields") bundles compatible versions of Eureka, Gateway, Config, OpenFeign, etc. for a specific Spring Boot minor version. |
| Service registry (Eureka) | A directory that services register themselves into on startup ("I'm `flight-service`, I'm at `10.0.0.5:8083`") and query at runtime to find each other, instead of hardcoding IPs/ports. |
| JWT (JSON Web Token) | A signed, self-contained token: a client presents it instead of re-sending credentials on every request. Anyone holding the signing key/secret can verify it wasn't tampered with, without a database lookup. |
| BCrypt | A deliberately slow password-hashing algorithm — the slowness is the point, it makes brute-forcing a stolen hash expensive. Never store plaintext or fast-hashed (e.g. plain SHA-256) passwords. |
| "Trust the gateway" | A pattern where only the API gateway validates JWTs; it forwards trusted identity headers (`X-User-Id`, `X-User-Roles`) to backend services, which trust those headers instead of re-parsing tokens. Only safe if backend services are unreachable except through the gateway — see ADR 0004. |
| OpenFeign | A declarative REST client: you write a Java interface with annotations describing an HTTP call, and Spring generates the actual HTTP call at runtime — including resolving the target service's address via Eureka. |
| `@PreAuthorize` | A Spring Security annotation that guards a method with a permission check (e.g. `hasRole('ADMIN')`), evaluated before the method body runs. Requires `@EnableMethodSecurity`. |
| `@OneToMany` / `@ManyToOne` | JPA relationship mapping — one `Hotel` has many `RoomType`s, each `RoomType` belongs to one `Hotel`. `cascade = ALL` + `orphanRemoval = true` means saving/deleting the Hotel saves/deletes its RoomTypes too, since they have no independent lifecycle. |
| Saga pattern | A way to handle a "transaction" that spans multiple services/databases, where no single database transaction can cover all of it. Instead of rollback, each step that already succeeded gets explicitly undone by a compensating action if a later step fails. |
| Orchestration vs. choreography (Saga) | Orchestration: one service (booking-service) directly calls every participant and issues compensating calls on failure — a central coordinator. Choreography: services react to each other's published events with no central coordinator — needs an event bus (Kafka, Phase 5). |
| Pessimistic write lock (`SELECT ... FOR UPDATE`) | Locks a row so no other transaction can read-and-modify it until the lock-holding transaction commits or rolls back. Only works within the DB connection/transaction that holds the lock — can't be held across an HTTP call to another service. |
| Idempotent (in this context) | Calling an operation twice with the same input has the same effect as calling it once. loyalty-service's `earn`/`reverse` check for an existing transaction record before acting, so a retried compensating call can't double-credit or double-refund. |
| Compensating action | The explicit "undo" for a step that already succeeded in a Saga — e.g. `releaseSeats` undoes `reserveSeats`, `refund` undoes `authorize`. Not a database rollback; a deliberate, separate operation. |
| Kafka topic / consumer group | A topic is a named event stream (e.g. `booking-events`) that producers publish to and consumers read from. A consumer group is a named set of consumers sharing the work of reading a topic — each message goes to one consumer *within* a group, but every group gets its own full copy of the stream. |
| `max.block.ms` | A Kafka producer setting most people never touch: how long `KafkaTemplate.send()` is allowed to block the *calling thread* to resolve topic metadata (partition info) before giving up. Defaults to 60000ms — surprising, since `send()` looks like a fire-and-forget async call that returns a `CompletableFuture`. If the broker is unreachable, that initial call can itself hang for a full minute before you even get the future back. |
| Circuit breaker | Wraps a remote call; after enough recent failures (a configurable failure-rate threshold over a sliding window), it "opens" and rejects further calls immediately without even attempting them, for a cooldown period — protects a struggling downstream service from being hammered further, and protects the caller from piling up slow/timed-out requests. |
| Idempotency-aware retry | Automatically retrying a failed call is only safe if calling it twice has the same effect as calling it once. In this project, retry is enabled only on `getFlight`/`getHotel` (read-only) and the guarded-idempotent `refund`/loyalty `earn`/`reverse` — never on `reserveSeats`/`authorize`, where a retried timeout could double-act on a request that actually succeeded server-side. |
| Trace ID / span ID | A trace ID identifies one logical request as it flows through every service it touches; a span ID identifies one hop within that trace (e.g. the call from booking-service to flight-service). Correct propagation means the *same* trace ID appears in every service's logs for one request, each with a *different* span ID. |
| Trace context propagation | How a trace ID gets from one service to the next — typically an HTTP header (`traceparent`) added to the outgoing request and read back out on the receiving end. This doesn't happen automatically for every HTTP client; Feign specifically needs the `feign-micrometer` dependency to hook into it — see ADR 0008. |
| Layered jar | Spring Boot's default jar layout split into layers (dependencies, snapshot-dependencies, spring-boot-loader, application) by how often each changes. `java -Djarmode=tools -jar app.jar extract --layers` unpacks them separately, so a Docker image can `COPY` the rarely-changing dependency layers first and the frequently-changing application layer last — better build-cache reuse than one monolithic `COPY app.jar`. |
| 12-factor config | Configuring an app entirely through environment variables (not baked-in profile files), so the exact same build artifact runs unmodified in every environment — only what's injected around it differs. Spring Boot's relaxed binding maps env vars like `SPRING_DATASOURCE_URL` straight onto `spring.datasource.url`. |
| Jackson default typing (polymorphic typing) | Jackson can embed a type hint (`@class` property, or an array-wrapper form) in serialized JSON so a later deserialization call — where the target type is `Object`/generic and otherwise unknown — can reconstruct the right concrete class. Powerful but finicky (inconsistent wrapper formats for "non-natural" scalar types like `BigDecimal`) and a known deserialization-gadget security risk if the JSON source isn't fully trusted — best avoided when a type-specific serializer is possible instead. |
| Redis cache serializer binding | Configuring a *specific* Jackson `JavaType` (e.g. `List<FlightResponse>`) for one named Spring cache, instead of relying on a generic serializer with embedded type metadata. Since a given `@Cacheable` cache name in this project only ever holds one well-known return type, this sidesteps default typing entirely — Jackson is simply told up front what to deserialize into. |
| Zipkin reporter | The component that actually ships completed spans to a Zipkin server over HTTP. `micrometer-tracing-bridge-brave` creates and correlates trace/span IDs (visible in log output) but does nothing to *export* them — a separate `zipkin-reporter-brave` dependency is required for spans to ever leave the process. Missing it is a silent failure: sampling and log correlation work fine, so nothing looks obviously broken until you check the Zipkin UI and find it empty. |
| kind (Kubernetes-in-Docker) | Runs a local Kubernetes cluster where each "node" is itself a Docker container, rather than a full VM (minikube's default). Lightweight and fast to create/destroy, at the cost of needing `kind load docker-image` to make a locally-built image visible inside the cluster — its nodes don't share the host's Docker image cache automatically. |
| Deployment / ReplicaSet / Pod | A Deployment declares desired state ("run N replicas of this pod spec"); it creates and owns a ReplicaSet, which creates and owns the actual Pods. If a Pod dies, the ReplicaSet notices and creates a replacement — the reconciliation loop made concrete. You almost never create a ReplicaSet or Pod directly; you declare a Deployment and let the chain do its job. |
| Service (Kubernetes) | A stable virtual IP + DNS name that load-balances across whichever Pods currently match its label selector. Pods are ephemeral and get new IPs when recreated; a Service is Kubernetes' own answer to the same "how do I find the current instance of X" problem Eureka solves at the application layer — worth sitting with as a real design question once both exist in the same system. |
| ConfigMap / Secret | Both inject configuration into a Pod (as env vars or mounted files); a Secret is only base64-encoded, not encrypted at rest by default, so it's not meaningfully more secure than a ConfigMap unless paired with a real secrets-management layer (Vault, sealed-secrets, a cloud provider's secrets manager). |
| NodePort | A Service type that opens a specific port (30000-32767 by default) on every cluster node, reachable from outside the cluster. In `kind`, reaching a NodePort from the host additionally requires an `extraPortMappings` entry in the cluster config — kind's Docker-based nodes aren't otherwise reachable from `localhost` at all. |
| startupProbe | Suppresses `livenessProbe`/`readinessProbe` checks entirely until it succeeds once, with its own separate (usually much longer) failure budget. Purpose-built for slow-starting containers — a JVM under CPU contention can take minutes to boot, and a `livenessProbe` with a short `initialDelaySeconds` will kill and restart it repeatedly before it ever finishes, which is exactly what happened live in Phase 8. |
| Thundering herd (redeploy) | Applying manifest changes to many Deployments at once, when the fix targets a shared resource-contention problem, can make the problem worse: each changed Deployment starts a *new* ReplicaSet/Pod without necessarily killing the old one first, so failing old Pods and failing new Pods now compete for the same constrained CPU/memory simultaneously. The fix is deploying in small batches, verifying each is stable (0 restarts) before adding the next — not just raising a timeout. |
| Helm chart | A packaged, parameterized set of Kubernetes manifests: templates (Go templating syntax over YAML) plus a `values.yaml` supplying the data that fills them in. `helm install`/`helm upgrade` renders every template with the current values and applies the result — the templating-and-packaging layer Kubernetes itself doesn't provide. |
| `helm template` / `helm lint` | Render a chart to plain YAML locally, or check it for structural issues — both run without touching any cluster. The way to catch a templating mistake (bad indentation, wrong conditional) before ever running `helm install`. |
| `range` (Helm/Go template) | Iterates over a list or map from `values.yaml` inside a template, emitting one copy of the enclosed YAML per item with that item's fields substituted in. What turns "9 nearly-identical Deployment+Service manifests" into "one template plus a 9-entry list of names/ports/flags." |
| Staged rollout (via values override) | Deploying only a subset of a chart's resources at first (here, via a `tier`/`maxTier` values field gating which templates render), then progressively enabling more via `helm upgrade --set maxTier=N`, waiting for each batch to stabilize before advancing. A manual workaround for a real resource-contention constraint, not Helm's normal behavior — the production-grade version of the same idea is Helm hooks with weighted Jobs, readiness gates, or GitOps sync waves. |
