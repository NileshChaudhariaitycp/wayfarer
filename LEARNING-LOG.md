# Learning Log

Per-phase comprehension checkpoints. Each phase adds a section once it's complete.

## Phase 0 ‚Äî Repo scaffold
- What is a Maven multi-module reactor, and why use one instead of 11 independent repos for this project?
- Why does Spring Cloud have its own "release train" version instead of matching Spring Boot's version number exactly?
- What would happen if `flight-service` and `api-gateway` ended up on different Spring Boot patch versions?

## Phase 1 ‚Äî Backbone (discovery-server, config-server, api-gateway)
- If `config-server` crashed right now, would `api-gateway` immediately stop routing to it ‚Äî or only fail on the *next* new request? What does that tell you about how `lb://` resolution actually works?
- Why does `spring.cloud.gateway.server.webflux.routes` use `lb://CONFIG-SERVER` instead of a hardcoded `http://localhost:8888`? What would break if we'd hardcoded it?

## Phase 2 ‚Äî Identity (auth-service, user-service, gateway JWT validation)
- Walk through what happens, service by service, from `POST /auth/login` to a subsequent authorized `GET /users/me` ‚Äî where is the JWT parsed, and where is it never seen again?
- Why does `user-service` return 403 (not 401) when a CUSTOMER requests someone else's profile via `/users/{id}`? What's the difference between the two status codes, semantically?
- The registration flow does a synchronous Feign call from auth-service to user-service *inside* the same request. What happens to the system's state if user-service is down at that exact moment? Is that acceptable, and why did we accept it for now instead of fixing it immediately?
- Right now, is anything actually stopping you from calling `user-service` on port 8082 directly, skipping the gateway and its JWT check entirely? Try it. What does that tell you about how much protection "trust the gateway" is really providing today, locally?

## Phase 3 ‚Äî Inventory (flight-service, hotel-service)
- Why does the gateway need a *method-aware* public-path check for `/flights/**` and `/hotels/**`, when `/auth/register` and `/auth/login` were fine with a plain path match?
- `RoomType.hotel` is `@ManyToOne(fetch = FetchType.LAZY)` but `Hotel.roomTypes` is `fetch = FetchType.EAGER`. What's the practical difference, and why might always-eager loading of a collection become a problem as hotel-service's data grows?
- `flight-service`'s `update()` recomputes `seatsAvailable` from `totalSeats - seatsBooked` instead of just overwriting it with the request's value. Why does that matter once real bookings exist?

## Phase 4 ‚Äî Orchestration (booking-service Saga, payment-service, loyalty-service, pessimistic locking)
- `BookingOrchestrationService` is deliberately NOT wrapped in one `@Transactional` spanning the whole Saga. Walk through what would go wrong if it were, specifically around the moment right after seats are reserved but before payment is attempted.
- During testing, a real "bug" turned out to be a test-data collision: an earlier manual call to loyalty-service's `/earn` endpoint used `bookingId=1`, and booking-service's first real booking also got `id=1`, so the idempotency check silently skipped crediting points. What does this tell you about `bookingId` as a dedup key across two independently-seeded systems, and would this scenario ever actually happen in production?
- `flight-service`'s `releaseSeats` is explicitly *not* idempotent (documented in a comment), while `loyalty-service`'s `earn`/`reverse` are. Why was that an acceptable trade-off here, and what would have to be true for it to stop being acceptable?
- Trace what happens if `payment-service` itself crashes (not declines ‚Äî actually goes unreachable) mid-authorize, after `booking-service` already reserved seats. What state is the system left in, and who or what would need to notice and fix it?

## Phase 5 ‚Äî Event-driven (Kafka, notification-service)
- A real bug surfaced during this phase: with no Kafka broker reachable, a booking request took 60+ seconds and then failed with a 500, even though the publish call was written as "fire and forget." Walk through *why* `KafkaTemplate.send()` ‚Äî which returns a `CompletableFuture` and looks async ‚Äî could still block the calling HTTP thread for a full minute. What two separate fixes did that turn into (see `BookingEventPublisher`'s javadoc and `max.block.ms` in `booking-service`'s `application.yml`)?
- Docker Desktop wouldn't start on this machine during this phase, so the Kafka producer‚Üíconsumer flow was verified for *graceful degradation* (bookings still succeed, notification-service retries its connection instead of crashing) but not for an actual message being produced and consumed end-to-end. What specific test would you run the moment Kafka is available, and what would you expect to see in `notification-service`'s logs?
- Why does `notification-service` define its own copy of `BookingEvent` instead of importing booking-service's class? What actually happens if the two classes' field names drift out of sync ‚Äî does anything catch that at compile time?

## Phase 6 ‚Äî Resilience (Resilience4j) and observability (tracing, metrics)
- Walk through why `reserveSeats` and `authorize` never get automatic retry, while `getFlight` and loyalty's `earn`/`reverse` do. What's the actual failure scenario that makes blind retry dangerous on the first group?
- The live circuit-breaker test showed attempts 1‚Äì2 taking 8‚Äì13 seconds each, then attempts 3‚Äì6 failing in ~0.6s once the breaker opened. Explain what changed between those two groups of attempts, concretely ‚Äî what is the circuit breaker doing differently.
- A real bug was found where `booking-service` and `flight-service` showed *different* trace IDs for the same request. What was actually missing, and how do you tell the difference between "trace propagation is broken" and "these two trace IDs just happen to share a prefix by coincidence"?
- `api-gateway`'s `/actuator/prometheus` worked immediately with no security config, while every other service's needed an explicit permit-all rule added. Why the difference?

## Phase 7 ‚Äî Containerization (Docker, Postgres, Redis) ‚Äî fully live-verified
- A real bug: `flight-service`'s Redis-cached search endpoint returned 200 on the *first* call (cache miss, straight from Postgres) but a 500 on the *second* call (cache hit) with `IllegalArgumentException: object is not an instance of declaring class`. Walk through why a cache **write** could succeed while the very next **read** of that exact value failed ‚Äî what did `GenericJackson2JsonRedisSerializer` actually store the first time, and what was missing for it to reconstruct a real `FlightResponse` rather than a `LinkedHashMap` on the way back out?
- The first fix attempt (`objectMapper.activateDefaultTyping(...)`) "fixed" the crash but produced JSON mixing two different type-hint formats in the same document ‚Äî `{"@class":"...FlightResponse", ...}` for the bean, but `["java.math.BigDecimal", 289.99]` for its `basePrice` field. Why does Jackson treat `BigDecimal` differently from `String`/`Integer`/`Boolean` under `DefaultTyping.NON_FINAL`, even though `BigDecimal` is itself a final class? The final fix abandoned default typing entirely in favor of binding a `Jackson2JsonRedisSerializer` to a specific `JavaType` per cache name ‚Äî why is "the cache only ever holds one known type" a stronger guarantee than "the JSON carries its own type metadata," and what security property does it buy on top of correctness?
- A second, unrelated real bug: Zipkin's `/api/v2/services` endpoint returned `[]` ‚Äî completely empty ‚Äî even though the log pattern showed real trace IDs and span IDs on every request, and `management.tracing.sampling.probability: 1.0` was correctly set via config-server. What was actually missing from every service's `pom.xml`, and why did trace/span ID generation keep working perfectly with it absent? What does that tell you about the difference between *creating* a trace and *exporting* one?
- Two separate rebuild-and-redeploy attempts of 10 services at once, run concurrently by `docker compose build`, both failed identically at the exact same Maven step (`maven-dependency-plugin`'s own transitive dependency resolution) with "Failed to read artifact descriptor" errors ‚Äî while building the exact same 10 services one at a time, sequentially, succeeded cleanly. What resource is 10 concurrent `mvn dependency:go-offline` processes plausibly contending for that a sequential build wouldn't be?
- After recreating all 10 containers, `docker compose ps` showed every one as "Up" within seconds, but requests routed through the gateway to `flight-service`/`hotel-service` returned 503 for nearly a minute afterward, then started succeeding with no further changes. What two separate things were actually still happening during that window ‚Äî one inside the JVM, one inside the gateway ‚Äî and which log lines would you check to tell them apart?

## Phase 8 ‚Äî Kubernetes (kind) + raw manifests ‚Äî fully live-verified
- The original plan was a 1-control-plane + 2-worker kind cluster (closer to a real cluster's topology). It crashed Docker Desktop outright once real workloads started (3 kind-node containers, each a full kubelet+containerd+control-plane, plus Postgres/Redis/Kafka/Zipkin/Prometheus plus 11 JVMs all starting at once). Why does a 3-node *local* cluster cost meaningfully more than "3x the resources of 1 node," and what did switching to a single-node cluster *not* cost us in terms of this phase's actual teaching goals (Deployments/Services/ConfigMaps/Secrets/Helm)?
- A real bug: `config-server` failed to start with `"you need to configure a uri for the git repository"` ‚Äî even though its own `application.yml` hardcodes `spring.profiles.active: native` to use the classpath config-repo backend, not git. Walk through exactly why an env var (`SPRING_PROFILES_ACTIVE=docker`, set via a ConfigMap shared by every service) beat a value already set in the jar's own `application.yml`, and why `docker-compose.yml` never hit this same bug even though it deploys the identical jar.
- After bumping `startupProbe`'s `failureThreshold` and re-applying all 9 domain services' Deployments at once, the cluster ended up with **two pods per service** (old and new) both failing, instead of one healthy pod each. What Kubernetes mechanic (rolling update strategy) caused a manifest change intended to *fix* a resource problem to instead *double* the number of competing JVMs? What would you check in `kubectl get pods` to catch this pattern early next time?
- The eventual fix wasn't a bigger `startupProbe` timeout ‚Äî it was deploying the 9 domain services in small batches (2-3 at a time), waiting for each batch to reach `0` restarts before deploying the next. Why does staggering matter here when `docker-compose.yml`'s `depends_on: condition: service_healthy` chains effectively did the same staggering automatically, but the initial all-at-once `kubectl apply` did not? What's the K8s-native way real deployments solve this (staged rollouts, readiness gates, ArgoCd sync waves) that a raw `kubectl apply -f` over many files doesn't give you for free?
